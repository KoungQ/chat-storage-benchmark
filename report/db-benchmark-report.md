# Chat Storage Benchmark Report

## 1. 배경

이 보고서는 DorumDorum 채팅 저장소 선택을 위해 `PostgreSQL vs MongoDB`를 비교한 결과를 정리한다.

채팅 메시지 데이터는 다음 특성을 가진다.

- append 중심으로 계속 누적된다.
- 단건 조회보다 최신 메시지 목록, 이전 메시지 페이지 조회가 더 중요하다.
- 쓰기 빈도는 높지만 사용자 경험은 조회 성능에 더 민감하다.
- 현재 스키마는 평평하고 단순하다.

따라서 이번 비교의 핵심은 "문서형 저장소가 쓰기에서 유리하더라도, 채팅 핵심 조회 시나리오까지 포함하면 여전히 더 나은 선택인지"를 확인하는 데 있다.

## 2. 가설

| 시나리오 | 가설 | 근거 |
|:---|:---|:---|
| H1 전체 조회 | PostgreSQL 우세 가능 | 평평한 스키마에서는 row 기반 전체 조회가 더 단순할 수 있다. |
| H2 PK 조회 | 큰 차이 없을 가능성 | 둘 다 기본 키 기반 단건 조회다. |
| H3 조건 조회 | 인덱스 유무 영향이 클 가능성 | 단일 필드 조건 조회는 인덱스 효과가 직접적이다. |
| H4 페이지 조회 | PostgreSQL 우세 가능 | 정렬 + 페이지네이션 비용 차이가 직접 드러난다. |
| H5 삽입 | MongoDB 우세 가능 | append 중심 메시지 쓰기와 잘 맞을 수 있다. |
| H6 수정 | MongoDB 우세 가능 | 실제 차이는 저장 엔진 구현에서 갈릴 수 있다. |
| H7 삭제 | MongoDB 우세 가능 | 실제 차이는 저장 구조와 인덱스 유지 비용에서 갈릴 수 있다. |

## 3. 측정 조건

- 데이터: 채팅 메시지 10만 건
- 실행: 각 시나리오 1회
- 1차: 인덱스 없음
- 2차: 인덱스 추가 후 재측정
- 인덱스:
  - PostgreSQL: `sender_id`, `room_id`, `created_at`
  - MongoDB: `senderId`, `roomId`, `createdAt`

## 4. 1차 결과

| 시나리오 | PostgreSQL avg | MongoDB avg | 승자 |
|:---|---:|---:|:---|
| 전체 조회 | 2.50 s | 3.59 s | PostgreSQL |
| PK 조회 | 21.3 ms | 18.5 ms | MongoDB |
| 조건 조회 | 58.9 ms | 1.21 s | PostgreSQL |
| 페이지 조회 | 329 ms | 886 ms | PostgreSQL |
| 삽입 | 1.47 ms | 0.787 ms | MongoDB |
| 수정 | 5.50 ms | 3.75 ms | MongoDB |
| 삭제 | 61.2 ms | 40.1 ms | MongoDB |

요약:

- 전체 평균 응답 시간: PostgreSQL `135ms`, MongoDB `421ms`
- 에러율: 둘 다 `0%`
- P95 평균: PostgreSQL `3.60s`, MongoDB `3.52s`
- P99 평균: PostgreSQL `3.64s`, MongoDB `3.62s`

해석:

- 채팅 핵심 read path인 조건 조회와 페이지 조회에서 PostgreSQL이 우세했다.
- MongoDB는 PK 조회와 쓰기 시나리오에서 더 빨랐다.
- 평균 응답 시간 기준으로는 PostgreSQL이 전체적으로 더 안정적이었다.

가설 검증:

- 적중: H1, H2, H3, H4, H5, H6, H7
- 1차에서는 시나리오별 가설이 모두 결과와 같은 방향으로 나타났다.

## 5. 2차 결과

| 시나리오 | PostgreSQL avg | MongoDB avg | 승자 |
|:---|---:|---:|:---|
| 전체 조회 | 2.16 s | 2.94 s | PostgreSQL |
| PK 조회 | 19.4 ms | 21.0 ms | PostgreSQL |
| 조건 조회 | 36.1 ms | 23.9 ms | MongoDB |
| 페이지 조회 | 193 ms | 931 ms | PostgreSQL |
| 삽입 | 0.836 ms | 1.01 ms | PostgreSQL |
| 수정 | 3.32 ms | 3.18 ms | MongoDB |
| 삭제 | 28.2 ms | 6.41 ms | MongoDB |

요약:

- 전체 평균 응답 시간: PostgreSQL `82.0ms`, MongoDB `213ms`
- 에러율: 둘 다 `0%`
- P95 평균: PostgreSQL `2.01s`, MongoDB `2.57s`
- P99 평균: PostgreSQL `2.02s`, MongoDB `2.60s`

해석:

- 인덱스 적용 후 두 저장소 모두 빨라졌지만, 전체 평균 응답 시간은 여전히 PostgreSQL이 앞섰다.
- 조건 조회는 MongoDB가 `23.9ms`까지 내려오며 역전했다.
- 페이지 조회는 PostgreSQL `193ms`, MongoDB `931ms`로 격차가 유지됐다.
- 삽입은 PostgreSQL이 앞섰고, 수정과 삭제는 MongoDB가 더 빨랐다.

가설 검증:

- 적중: H1, H3, H4, H6, H7
- 빗나감: H2, H5

## 6. 종합 비교

| 지표 | PostgreSQL 1차 | MongoDB 1차 | PostgreSQL 2차 | MongoDB 2차 |
|:---|---:|---:|---:|---:|
| 전체 평균 응답 시간 | 135 ms | 421 ms | 82.0 ms | 213 ms |
| 평균 에러율 | 0% | 0% | 0% | 0% |
| P95 평균 | 3.60 s | 3.52 s | 2.01 s | 2.57 s |
| P99 평균 | 3.64 s | 3.62 s | 2.02 s | 2.60 s |
| 최대 응답 시간 | 9.50 s | 10.8 s | 7.56 s | 5.54 s |

종합 해석:

- 1차, 2차 모두 PostgreSQL이 전체 평균 응답 시간에서 우세했다.
- MongoDB는 쓰기 시나리오에서 반복적으로 강점을 보였다.
- 그러나 채팅 서비스 판단에서 더 중요한 페이지 조회는 두 차례 모두 PostgreSQL이 명확히 앞섰다.
- 조건 조회는 인덱스 적용 후 MongoDB가 더 빨라졌지만, 이 한 항목만으로 전체 선택이 뒤집히지는 않았다.

## 7. 결론

이번 비교의 최종 결론은 `PostgreSQL 유지`다.

첫째, 두 차례 측정 모두에서 PostgreSQL이 전체 평균 응답 시간 우위를 유지했다. 1차는 `135ms vs 421ms`, 2차는 `82.0ms vs 213ms`로, 인덱스 적용 전후를 막론하고 MongoDB보다 더 낮은 평균 응답 시간을 보였다. 이는 특정 시나리오 하나의 우연한 승패가 아니라, 전체 요청 흐름에서 PostgreSQL이 더 안정적인 응답 특성을 보였다는 뜻이다.

둘째, 채팅 서비스에서 더 중요한 읽기 시나리오, 특히 페이지 조회에서 PostgreSQL 우위가 일관됐다. 1차는 `329ms vs 886ms`, 2차는 `193ms vs 931ms`였다. 사용자가 실제로 가장 자주 체감하는 것은 "이전 메시지 불러오기", "최근 메시지 목록 읽기" 같은 페이지 조회다. 이 구간에서 격차가 유지됐다는 점이 이번 판단에서 가장 큰 근거다.

셋째, MongoDB의 강점도 분명했다. PK 조회, 수정, 삭제는 두 차례 중 적어도 한 번 이상 MongoDB가 앞섰고, 1차 삽입도 MongoDB가 더 빨랐다. 특히 2차에서 조건 조회가 `23.9ms`까지 개선되며 PostgreSQL `36.1ms`를 앞선 점은 인덱스 적용 시 MongoDB가 단일 필드 검색에서 충분히 경쟁력이 있음을 보여준다. 다만 이 강점은 주로 일부 쓰기 시나리오와 단일 조건 조회에 집중되어 있었다.

넷째, 운영 관점까지 포함하면 PostgreSQL 선택이 더 합리적이다. 이번 테스트에서 두 저장소 모두 1차, 2차 에러율이 `0%`였기 때문에, 의사결정 기준은 장애 대응보다 성능 특성과 운영 단순성에 맞춰도 된다. 이런 상황에서 MongoDB를 별도로 도입하면 저장소 이원화, 백업 전략 분리, 모니터링 포인트 증가 같은 운영 비용이 생긴다. 그런데 그 비용을 감수할 만큼 페이지 조회나 전체 평균 응답 시간에서 MongoDB가 우위를 보이진 않았다.

따라서 현재 DorumDorum의 채팅 저장소 전략은 `PostgreSQL 단일 채택`이 가장 타당하다. 이번 벤치마크는 MongoDB가 일부 쓰기 시나리오에서 더 빠를 수는 있지만, DorumDorum 채팅 서비스의 핵심 사용 패턴까지 포함하면 PostgreSQL이 더 실용적이라는 결론을 보여준다. 여기에 PostgreSQL과 MongoDB를 함께 운영할 때 발생하는 이종 DB 운영 복잡도까지 고려하면, 최종 선택은 `PostgreSQL 단일 운영`이 가장 합리적이다.

물론 MongoDB가 가지는 문서 구조의 유연성 자체는 분명한 장점이다. 다만 현재 DorumDorum의 채팅 메시지는 필드 구성이 단순하고 메시지 타입도 제한적이어서, 그 유연성이 지금 당장 저장소 선택을 바꿀 정도의 결정적 이점으로 이어지지는 않았다. 추후 메시지 타입이 다양해지거나 payload 구조를 더 유연하게 다뤄야 하는 요구사항이 생긴다면, 그 시점에는 `PostgreSQL JSONB` 같은 대안도 함께 검토할 수 있다.
